# pull-request 스킬

## 트리거

사용자가 다음 중 하나를 요청하면 이 스킬을 사용한다:
- PR 만들어줘 / PR 올려줘 / pull request 작성
- 리뷰 요청 준비해줘
- 브랜치 합치기 전에 PR 열어줘

## 핵심 원칙

1. **전체 커밋 히스토리 기준**: 최신 커밋 하나만 보지 않고, base 브랜치에서 갈라진 전체 변경을 파악한다.
2. **제목은 50자 이내**: 무엇이 달라졌는지 한 줄로 설명한다.
3. **본문은 WHY 중심**: 왜 이 변경이 필요했는지를 적는다. 무엇을 바꿨는지는 diff가 대신한다.

## PR 본문 구조

```markdown
## 요약
- (변경 핵심을 bullet로 1-3개)

## 변경 이유
(왜 이 작업이 필요했는지 1-3문장)

## 테스트 방법
- [ ] (검증 항목 1)
- [ ] (검증 항목 2)

## 관련 이슈
closes #<번호> (해당되는 경우)
```

## 워크플로우

1. `git log [base]...HEAD`로 포함될 커밋 전체 확인
2. `git diff [base]...HEAD`로 전체 변경 파악
3. 변경의 의도와 맥락 파악 후 제목과 본문 초안 작성
4. 현재 브랜치가 remote에 push되어 있는지 확인
5. 없으면 `git push -u origin <branch>` 먼저 수행
6. `gh pr create` 실행

## gh pr create 호출 형식

```bash
gh pr create \
  --title "제목" \
  --body "$(cat <<'EOF'
## 요약
...
EOF
)"
```

## 주의

- base 브랜치를 잘못 지정하면 엉뚱한 diff가 포함된다. 항상 사용자에게 base 브랜치를 확인한다.
- draft PR로 열지 일반 PR로 열지 사용자에게 묻는다.
- reviewer, label, milestone은 사용자가 명시한 경우에만 추가한다.

## 금지

- PR을 merge하지 않는다 (사용자가 명시적으로 요청해도 확인 절차를 반드시 거친다)
- force-push로 base 브랜치를 덮어쓰지 않는다
