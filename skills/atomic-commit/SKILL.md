# atomic-commit 스킬

## 트리거

사용자가 다음 중 하나를 요청하면 이 스킬을 사용한다:
- 커밋해줘 / 커밋 만들어줘 / commit
- 변경사항 정리해서 올려줘
- staged 변경을 커밋으로 나눠줘

## 핵심 원칙

1. **secret guard 먼저**: `.env`, 비밀번호, API 키가 스테이징되어 있으면 즉시 멈추고 사용자에게 알린다.
2. **atomic 단위로 분리**: 논리적으로 무관한 변경은 별도 커밋으로 나눈다.
3. **push는 사용자가 명시한 경우에만**: 커밋까지만 하고, push는 사용자가 동의하면 수행한다.

## 커밋 메시지 포맷

```
<type>(<scope>): <한국어 또는 영어 설명>

<선택: 변경 이유나 배경 1-2문장>
```

### type 목록
| type | 쓰는 경우 |
|------|-----------|
| feat | 새 기능 추가 |
| fix | 버그 수정 |
| refactor | 동작 변경 없는 코드 정리 |
| docs | 문서만 변경 |
| test | 테스트 추가/수정 |
| chore | 빌드, 설정, 의존성 |
| perf | 성능 개선 |
| ci | CI/CD 파이프라인 |

### scope
- 변경된 도메인이나 모듈을 괄호 안에 간략히 적는다.
- 전체 프로젝트에 걸친 변경이면 scope를 생략한다.

## 워크플로우

1. `git diff --cached`로 스테이징된 변경 확인
2. secret guard: 민감 패턴(`.env`, `password`, `api_key`, `secret`, `token`) 검색
3. 변경을 논리 단위로 그룹화
4. 각 그룹에 적합한 type과 설명으로 커밋 메시지 작성
5. 사용자에게 메시지 목록 보여주고 확인 받기
6. 확인 후 커밋 실행 (push는 사용자 지시 대기)

## 금지

- `--no-verify` 플래그 사용 금지 (사용자가 명시적으로 요청해도 이유를 먼저 묻는다)
- 스테이징되지 않은 파일을 임의로 `git add` 하지 않는다
- `.env`, 인증서, 토큰 파일을 커밋에 포함하지 않는다

## 출력 형식

커밋 전에 다음 형식으로 예고한다:

```
커밋 계획:
1. feat(auth): 로그인 토큰 갱신 로직 추가
2. test(auth): 토큰 갱신 단위 테스트

진행할까요?
```
